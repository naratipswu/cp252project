# เจาะลึกกระบวนการ CI/CD (Deep Dive into CI/CD Process)

เอกสารฉบับนี้อธิบายรายละเอียดเชิงลึกเกี่ยวกับระบบอัตโนมัติ (Automation) ในการพัฒนาโปรเจกต์ Camera Rental System ตั้งแต่การส่งโค้ดไปจนถึงการเตรียมความพร้อมก่อนการนำระบบขึ้นใช้งานจริง

---

## 🏗️ 1. ภาพรวมของระบบ (System Overview)

เราใช้แนวคิด **Automated Pipeline** โดยมี **GitHub Actions** เป็นหัวใจหลักในการกำกับดูแลขั้นตอนต่างๆ เพื่อให้มั่นใจว่าโค้ดทุกบรรทัดที่ถูกส่งเข้าสู่ Repository กลาง (Branch: `main`, `develop`) จะต้องผ่านเกณฑ์มาตรฐานด้านคุณภาพ ความปลอดภัย และประสิทธิภาพ

---

## 🚀 2. กระบวนการ Continuous Integration (CI)

CI คือการรวบรวมโค้ดและทดสอบความถูกต้องโดยอัตโนมัติ แบ่งออกเป็น 6 ขั้นตอนหลัก:

### ขั้นที่ 1: Code Quality Check (การตรวจสอบคุณภาพโค้ด)
- **เครื่องมือ**: ESLint
- **การทำงาน**: เมื่อนักพัฒนา Push โค้ด ระบบจะรันการสแกนหาสไตล์การเขียนโค้ดที่ผิดพลาด หรือ Syntax ที่อาจก่อให้เกิด Bug (Static Analysis)
- **เป้าหมาย**: รักษามาตรฐานการเขียนโค้ดให้เหมือนกันทั้งทีม (Coding Standard)

### ขั้นที่ 2: Unit Testing (การทดสอบระดับตรรกะ)
- **เครื่องมือ**: Jest / Istanbul
- **การทำงาน**: รันชุดการทดสอบย่อยๆ เพื่อตรวจสอบฟังก์ชันการคำนวณ (เช่น การคิดค่าปรับ, การคำนวณราคารวม)
- **Coverage**: ระบบจะบังคับตรวจสอบ Coverage (เปอร์เซ็นต์พื้นที่โค้ดที่ถูกทดสอบ) หากต่ำกว่าที่กำหนด Build จะล้มเหลว (Fail)

### ขั้นที่ 3: Building & Verification (การประกอบร่างแอปพลิเคชัน)
- **การทำงาน**: ระบบจะจำลอง Environment ใหม่ขึ้นมา (Clean Environment) และทำการติดตั้ง Dependencies ทั้งหมด (npm ci)
- **เป้าหมาย**: ตรวจสอบว่าแอปพลิเคชันสามารถ Start ขึ้นมาได้จริงโดยไม่มีปัญหาเรื่อง Library หายหรือ Config ผิดพลาด

### ขั้นที่ 4: E2E UI Testing (การทดสอบหน้าบ้านเต็มระบบ)
- **เครื่องมือ**: Cypress
- **การทำงาน**: รันชุดทดสอบที่จำลองพฤติกรรมผู้ใช้จริง (Simulation) เช่น การกรอกฟอร์มการจองกล้อง จนไปถึงการจ่ายเงิน
- **จุดเด่น**: หากเทสล้มเหลว ระบบจะทำการ **Snapshot และบันทึกวีดีโอ** ทิ้งไว้ให้ใน Artifacts เพื่อให้โปรแกรมเมอร์ดูย้อนหลังได้ว่าหน้าจอพังที่ตรงไหน

### ขั้นที่ 5: Security Scanning (การสแกนความปลอดภัย)
- **เครื่องมือ**: `npm audit` และ OWASP Dependency Check
- **การทำงาน**: ตรวจหา Libraries ที่ล้าสมัยหรือมีช่องโหว่ทางความปลอดภัย (Vulnerabilities) ที่เป็นอันตรายต่อระบบ

### ขั้นที่ 6: Performance Profiling (การวัดประสิทธิภาพ)
- **เครื่องมือ**: Autocannon
- **การทำงาน**: ทำ Stress Test จำลองโหลดเพื่อดูว่าโค้ดใหม่ทำให้ระบบช้าลงหรือไม่ (Latency Regression)

## 🚚 3. กระบวนการ Continuous Deployment (CD)

เมื่อขั้นตอน CI ทั้งหมดผ่าน (Passed) ระบบจะเข้าสู่กระบวนการเตรียมความพร้อมเพื่อใช้งานจริง:

### การจัดการ Environments
1.  **Staging Environment**: โค้ดจาก Branch `develop` จะถูกเตรียมไว้เพื่อทดสอบในสภาพแวดล้อมที่เหมือนจริงที่สุด
2.  **Production Environment**: โค้ดจาก Branch `main` ที่ผ่านการรีวิว (Pull Request) จะถูกเตรียมเพื่อ Deploy สู่โฮสต์จริง

### Deployment Strategy
- **Manual Gate**: สำหรับการนำขึ้น Production จริง เรามีการใช้เงื่อนไขตรวจสอบ (Condition) เพื่อให้ Admin กดอนุมัติก่อน (Manual Approval) ในบางขั้นตอน เพื่อป้องกันความผิดพลาด
- **Secrets Management**: เราเก็บค่า Config สำคัญ (เช่น DB Password, API Keys) ไว้ใน **GitHub Secrets** เพื่อความปลอดภัย ไม่มีการฝังรหัสไว้ในโค้ดตรงๆ

---

## ⚡ 4. เทคนิคการเพิ่มความเร็ว (Pipeline Optimization)

เพื่อให้ Pipeline ทำงานได้รวดเร็ว (ไม่ให้โปรแกรมเมอร์รอนาน) เราได้ใช้เทคนิค:
1.  **Parallel Matrix**: แยกการรันเทสประสิทธิภาพออกเป็นหลายส่วนและทำงานพร้อมกันในเวลาเดียว
2.  **Caching**: ทำการจดจำโฟลเดอร์ `node_modules` ไว้ในระบบ Cache เพื่อให้การรันครั้งต่อไปไม่ต้องโหลดใหม่ทั้งหมด
3.  **Dockerized Databases**: ใช้ Postgres Container ใน Pipeline เพื่อจำลองฐานข้อมูลที่สดใหม่เสมอสำหรับการเทสแต่ละครั้ง

---

## ✅ 5. สรุปประโยชน์ของระบบ CI/CD นี้
- **ลด Human Error**: ไม่ต้องฝากความหวังว่าโปรแกรมเมอร์จะเทสเองครบไหม เพราะระบบบังคับเทสให้ทุกครั้ง
- **เร็วและแม่นยำ**: การรันเทสขนานกันทำให้รู้ผลไวภายในไม่กี่นาที
- **ความมั่นใจสูง**: ทุกการแก้ไขจะถูกการันตีด้วยหลักฐาน (Reports/Videos) ว่าใช้งานได้จริง 100% ก่อนถึงมือลูกค้า

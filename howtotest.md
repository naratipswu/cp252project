# คู่มือการทดสอบและกระบวนการทำงาน (How to Test & Process)

เอกสารฉบับนี้อธิบายถึงขั้นตอนการทดสอบระบบและกระบวนการทำงานในระยะที่ 4 เพื่อให้ทีมงานเข้าใจวิธีการรันเทสและอ่านค่าผลลัพธ์ต่างๆ ได้อย่างถูกต้อง

---

## 🛠 1. วิธีการรันการทดสอบ (How to Run Tests)

เราได้เตรียมคำสั่งสำหรับการทดสอบไว้ใน `package.json` แล้ว โดยแบ่งออกเป็น 3 ประเภทหลัก:

### 1.1 การทดสอบระดับตรรกะ (Unit Testing)
ใช้สำหรับดักจับข้อผิดพลาดในระดับ Code Logic (เช่น การคำนวณเงิน)
- **คำสั่ง**: `npm test` หรือ `npm run test:cov` (เพื่อดู Coverage)
- **เครื่องมือ**: Jest / Istanbul

### 1.2 การทดสอบระดับผู้ใช้ (UI / E2E Testing)
ใช้สำหรับจำลองการคลิกหน้าเว็บจริงแบบ End-to-End
- **คำสั่ง**: `npm run test:ui`
- **เครื่องมือ**: Cypress (รันแบบ Headless)
- **สิ่งที่คาดหวัง**: ต้องผ่านครบทั้ง 5 Test Cases (สมัครสมาชิก, ค้นหา, ดูข้อมูล, เลือกวันที่, จองลงตะกร้า)

### 1.3 การทดสอบประสิทธิภาพ (Performance Profiling)
ใช้สำหรับวัดความเร็วและความอึดของ Server
- **คำสั่ง**: `npm run profile`
- **เครื่องมือ**: Autocannon
- **ไฟล์ผลลัพธ์**: ดูได้ที่ `profiling.md` และ JSON ในโฟลเดอร์ `performance/results/`

## 📈 2. การอ่านค่า Profiling (Understanding Metrics)

ค่าต่างๆ ที่ปรากฏใน `profiling.md` มีความหมายดังนี้:

| ค่าที่วัด (Metric) | ความหมายในภาษาไทย | ความสำคัญ |
|:---:|:---|:---|
| **Requests** | จำนวนครั้งที่เรียกใช้งาน | บอกถึงปริมาณข้อมูลที่ระบบรับมือได้ในระยะเวลาที่กำหนด |
| **Throughput** | ความเร็วในการส่งข้อมูล (Req/s) | บอกว่า Server หนึ่งเครื่องรับ User ได้กี่คนต่อวินาที (ยิ่งสูงยิ่งดี) |
| **Latency Mean** | ค่าความช้าเฉลี่ย (ms) | เวลาเฉลี่ยที่ User ต้องรอก่อนจะเห็นหน้าเว็บ (ยิ่งน้อยยิ่งดี) |
| **Latency P99** | ค่าความช้าสูงสุดที่เปอร์เซ็นไทล์ 99 | บอกถึง "ประสบการณ์ที่แย่ที่สุด" ที่ User มีโอกาสเจอ (ควรคุมไม่ให้สูงโดด) |

## 🔄 3. กระบวนการทำงาน (Project Process)

เราใช้กระบวนการทำงานแบบ **Automated Lifecycle** เพื่อลด Human Error:

1.  **Code & Lint**: เขียนโค้ดและตรวจสอบ Syntax ด้วย ESLint
2.  **Push & Trigger CI**: เมื่อส่งโค้ดขึ้น GitHub ระบบ GitHub Actions จะรัน Pipeline อัตโนมัติ
3.  **Verify & Test**: ระบบจะรัน Unit Test และ Cypress ให้ทุกครั้ง เพื่อให้มั่นใจว่าของใหม่ไม่ไปพังของเก่า
4.  **Profile & Optimize**: ทำการวัดความเร็ว (Profiling) เพื่อดูว่ามีจุดไหนของ Code ที่ทำให้ Server ช้าลงอย่างผิดปกติหรือไม่
5.  **Evidence Collection**: เก็บหลักฐานการทดสอบไว้ใน Markdown (เช่น `Ui_test.md`) เพื่อใช้ประกอบการทำรายงานสรุปผล

---
**สรุป**: การทดสอบที่ดีต้องครอบคลุมทั้ง **ความถูกต้อง (Function)**, **ความเร็ว (Speed)**, และ **ความปลอดภัย (Security)**
